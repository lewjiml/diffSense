# AI 初审需求覆盖度扫描 —— 操作指南

> 本指南演示「需求拆解 → 代码扫描 → 覆盖度报告」的完整流程。
> 配套文件：
> - 需求文档：`demo/requirements.md`（基于「案例展示网站」真实代码反推）
> - 被扫描代码：**T 盘原项目** `T:\我的Claw本地知识库\04-副业落地\案例展示网站\backend\src`

> 说明：本 demo 的需求文档完全对应你的案例展示网站后端实现，跑完 scan 应该得到 **100% 覆盖率**，
> 这正是「初始版本」的理想状态。第二轮我们再加需求、改代码，就能看到覆盖率掉下来的效果。

## 一、第一步：拆解需求文档（parse）

### 命令

在项目根目录 `ai-initial-review-demo/` 下执行（**拆全部 9 个板块，不传 `--section`**）：

```powershell
node ai-req.js parse demo/requirements.md --module Showcase
```

> 💡 不传 `--section` 时，工具会处理文档里**所有 `##` 板块**，一次性拆出全部需求条目，
> 适合本 demo 这种「整站需求」场景。

### 参数说明

| 参数 | 含义 |
|------|------|
| `demo/requirements.md` | 需求文档路径 |
| `--module Showcase` | 输出文件名后缀，生成 `requirements-Showcase.json` |

### 预期输出

```

══════════════════════════════════════════════════════
```

### 产物

- **`requirements-Showcase.json`** —— 结构化需求条目，每条带 `id`（REQ-001 ~ REQ-XX）、`title`、`description`、`priority`、`category`、`acceptance` 字段。

> 💡 **可选玩法**：只拆某个板块（比如只看鉴权部分）：
> ```powershell
> node ai-req.js parse demo/requirements.md --section "管理员鉴权" --module Auth

## 二、第二步：扫描代码覆盖度（scan）

### 命令（扫描 T 盘原项目后端代码）

```powershell
node ai-req.js scan --req requirements-Showcase.json --dir "T:\我的Claw本地知识库\04-副业落地\案例展示网站\backend\src"
```

### 参数说明

| 参数 | 含义 |
|------|------|
| `--req requirements-Showcase.json` | 上一步生成的需求文件 |
| `--dir "<T盘路径>\backend\src"` | 代码来源：读取该目录下所有代码文件 |

> ⚠️ **路径包含中文与空格，必须用双引号包起来**，否则 PowerShell 会拆参数。

代码来源三选一（本 demo 用 `--dir` 最简单）：

| 参数 | 含义 | 适用场景 |
|------|------|----------|
| `--dir <目录>` | 读取目录下所有代码 | demo 演示、本地全量扫描 |
| `--base <分支>` | `git diff <base>...HEAD` | 接了 git 的真实项目 |
| `--diff <文件>` | 从 diff 文件读取 | MR/PR 场景 |

### 预期输出（初始版本，所有需求都应被覆盖）

## 四、进阶：需求过滤

`scan` 支持几个过滤参数，只扫描部分需求：

```powershell
# 只扫 P0 需求（核心功能）
node ai-req.js scan --req requirements-Showcase.json --dir "T:\...\backend\src" --priority P0

# 只扫指定 ID
node ai-req.js scan --req requirements-Showcase.json --dir "T:\...\backend\src" --include-ids REQ-001,REQ-004,REQ-010

# 排除某些 ID（比如跳过前端页面类需求）
node ai-req.js scan --req requirements-Showcase.json --dir "T:\...\backend\src" --exclude-ids REQ-020,REQ-021

# 只扫 enabled !== false 的需求
node ai-req.js scan --req requirements-Showcase.json --dir "T:\...\backend\src" --only-enabled
```

> 💡 **实用技巧**：前端页面类需求（REQ-020 ~ REQ-027）在后端代码里扫描意义不大，
> 可以在 `requirements-Showcase.json` 里把这些条目的 `enabled` 字段改为 `false`，
> 然后用 `--only-enabled` 只扫后端能覆盖的需求，这样演示效果最干净。

## 六、下一步预告

当你完成上面的初始扫描后，告诉我「**跑通了**」。

我会做第二轮：
1. 在 `requirements.md` 里**新增需求**（比如「访客留言提交接口」「作品访问统计」「批量上下架」等代码里没有的功能）
2. **改动代码**（可能新增部分实现，也可能故意留几条不实现）
3. 你重新跑一次 scan，就能看到 AI 精准识别出**「新增需求未被覆盖」**的效果——覆盖率从 100% 掉下来，报告里会清晰列出未覆盖项

这才是 AI 初审工具的核心价值：**在代码合并前发现漏实现的需求**。

## 常见问题

**Q：报错 `未配置 AI_API_KEY`？**
A：检查项目根目录的 `.env` 文件是否存在、`AI_API_KEY` 是否填了。

**Q：报错 `请求超时`？**
A：思考模型响应较慢，工具默认超时 180 秒，请耐心等待，或换用响应更快的模型。

**Q：扫描 30+ 条需求太慢？**
A：先用 `--priority P0` 只扫核心需求；或在 `requirements-Showcase.json` 里把前端类需求 `enabled:false`。

**Q：覆盖率不准确？**
A：可以调整需求描述的措辞让它更明确，或在 `requirements-Showcase.json` 里补充 `acceptance` 验收标准。

**Q：路径里有中文报错？**
A：Windows PowerShell 下路径必须用双引号包起来，例如 `"T:\我的Claw本地知识库\..."`。
